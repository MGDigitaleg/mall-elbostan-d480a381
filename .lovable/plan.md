# خطة تطوير “عروض الافتتاح” داخل البنية الحالية

## ما سيتم بناؤه
سيتم تحويل مسار العروض الحالي من حالة placeholder إلى تجربة متكاملة داخل منظومة مول البستان، مع الاعتماد على الجداول والمسارات الموجودة بالفعل، وليس إنشاء نظام منفصل أو microsite جديد.

## التنفيذ
1. **إعادة استخدام بنية العروض الحالية**
   - الإبقاء على جدول `deals` كقاعدة العروض الحالية وربط كل عرض بمتجر عبر `store_id`.
   - الإبقاء على `/daily-deals` كصفحة العروض الرئيسية، لكن تحويلها من عدّاد فقط إلى صفحة “عروض الافتتاح”.
   - الإبقاء على `DealsTeaser` في الصفحة الرئيسية وتطويره ليعرض subset من نفس البيانات المنظمة.
   - الإبقاء على ربط العروض في Atrium/Hub مع تحسين الاستعلامات والعرض بحيث يستخدم نفس البطاقات/البيانات المنظمة قدر الإمكان.

2. **إضافة البيانات المنظمة اللازمة للعروض ومتاجر الافتتاح**
   - تحديث بيانات المتاجر الموجودة بدل إنشاء كيان منفصل:
     - `Infinity` → تحسين الاسم والعربية/الوصف/التصنيف ليصبح `Infinity Computer Services / ICS`.
     - `Kareem Store` → تحسين الاسم/الوصف/التصنيف ليتوافق مع سياق عروض الافتتاح.
   - إدخال عروض ICS وKareem داخل جدول `deals` وربطها بالمتاجر الحالية.
   - استخدام حقل `description_ar` وحقول موجودة + `promo_code` عند الحاجة، مع إضافة حقول منظمة جديدة فقط إذا كانت ضرورية لعرض:
     - السعر الحالي
     - السعر السابق
     - المواصفات المختصرة
     - صورة العرض/المنتج
     - شارة العرض
     - حالة الافتتاح/يفتتح قريبًا
     - حالة التحقق
   - إذا كانت الحقول الحالية غير كافية، سيتم توسيع **الجدول الحالي `deals`** بدل إنشاء offers system جديد.

3. **تحويل `/daily-deals` إلى صفحة عروض افتتاح فعلية**
   - قبل 1 مايو 2026:
     - Hero بعنوان قوي + countdown
     - Featured opening offer أو featured merchant block
     - preview grid حقيقي للعروض أسفل الـ hero
     - فلاتر بسيطة حسب المتجر أو الفئة
     - إبراز “يفتتح قريبًا”
   - بعد 1 مايو 2026:
     - التحول تلقائيًا إلى grid/list كامل للعروض الحية
     - إزالة الاعتماد على placeholder-only experience
   - تحسين SEO والعناوين والنسخ العربية لتصبح الصفحة صالحة للفهرسة عندما تحتوي على محتوى فعلي.

4. **تطوير صفحة العروض/المدخل الحالي للعروض**
   - تحسين بطاقة “عروض الافتتاح” داخل الـ hub/teaser لتبدو نشطة وليست static promo box.
   - إضافة مؤشرات حقيقية من البيانات الحالية مثل:
     - عدد المحلات المشاركة
     - عدد العروض الحالية
     - عدد عروض الافتتاح القادمة
   - جعل CTA واضحًا ويقود إلى `/daily-deals` باعتبارها الوجهة الأساسية للعروض.

5. **ربط العروض بصفحات المتاجر**
   - إضافة قسم في صفحة المتجر:
     - `عروض هذا المحل` أو `عروض الافتتاح من هذا المحل`
   - يظهر للمتاجر المشاركة فقط.
   - استخدام نفس مكوّن البطاقة reusable مع نسخة compact مناسبة لصفحة المتجر.
   - إضافة روابط متبادلة:
     - من العرض إلى صفحة المتجر
     - ومن صفحة المتجر إلى جميع عروض الافتتاح أو جميع عروض هذا المتجر

6. **إضافة teaser محدود على الصفحة الرئيسية**
   - تطوير `DealsTeaser` ليصبح teaser فعلي لعروض الافتتاح بدل مزج عام/مبهم.
   - عرض عدد محدود من أفضل العروض فقط.
   - الحفاظ على توازن الصفحة الرئيسية حتى لا تتحول إلى صفحة عروض مزدحمة.

7. **تصميم بطاقات عروض موحدة وقابلة لإعادة الاستخدام**
   - بناء مكوّن OfferCard/MerchantOfferCard موحد للاستخدام في:
     - الصفحة الرئيسية
     - صفحة `/daily-deals`
     - صفحة المتجر
     - ويمكن لاحقًا استخدامه داخل الـ atrium offers tab
   - البطاقة ستتضمن:
     - شعار أو اسم المتجر
     - شارة `يفتتح قريبًا` عند الحاجة
     - صورة المنتج
     - اسم المنتج
     - مواصفات مختصرة
     - السعر الحالي
     - السعر السابق مع strike-through
     - نسبة/قيمة الخصم
     - الفئة
     - CTA لصفحة المتجر
     - CTA ثانوي عند الحاجة لعرض كل عروض هذا المتجر

## تفاصيل تقنية
- **الملفات الحالية التي سيتم البناء عليها**
  - `src/pages/DailyDeals.tsx`
  - `src/components/home/DealsTeaser.tsx`
  - `src/components/home/HomeContent.tsx`
  - `src/pages/StoreDetail.tsx`
  - `src/components/map/AtriumHubModal.tsx`
  - `src/pages/admin/AdminPages.tsx`

- **البيانات الحالية التي سيتم إعادة استخدامها**
  - جدول `deals`
  - جدول `stores`
  - الربط الحالي `deals.store_id -> stores`
  - المتجران الموجودان بالفعل:
    - `Infinity` (`slug: infinity`)
    - `Kareem Store` (`slug: kareem-store`)

- **تغييرات قاعدة البيانات المتوقعة**
  - فقط إذا لزم الأمر لعرض العروض بشكل structured فعلاً، سيتم توسيع جدول `deals` بدل إنشاء جدول عروض منفصل.
  - أمثلة الحقول المرشحة عند الحاجة:
    - `offer_title_ar`
    - `brand`
    - `model`
    - `specs_short_ar`
    - `price_current`
    - `price_old`
    - `currency`
    - `offer_badge_ar`
    - `image_primary`
    - `source_link`
    - `source_type`
    - `verified`
    - `opening_status`
  - سيتم الإبقاء على سياسات الوصول العامة الحالية للعروض الحية، مع عدم إضافة أي كشف لبيانات حساسة.

- **منطق العرض الزمني**
```text
Before launch:
  Hero + Countdown + Preview offers + Opening soon messaging

On/after launch:
  Full opening offers listing
```

- **النسخ العربية**
  - عروض الافتتاح
  - عروض المحلات الجديدة
  - يفتتح قريبًا
  - عرض خاص بمناسبة الافتتاح
  - شاهد كل عروض هذا المحل
  - عرض ضمن افتتاح مول البستان
  - جميع عروض الافتتاح

## النتيجة النهائية
سيصبح نظام العروض جزءًا طبيعيًا من تجربة مول البستان:
- نفس المتاجر
- نفس صفحات المتاجر
- نفس الصفحة الرئيسية
- نفس صفحة العروض الحالية بعد تطويرها
- دون microsite منفصل
- ودون gallery فوضوية من بوستات السوشيال
- مع تجربة عربية أنيقة ومتكاملة وقابلة للتوسع بعد الافتتاح
